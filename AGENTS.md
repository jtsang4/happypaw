# AGENTS.md

## 目标

本项目默认遵循对 Coding Agent 友好的组织原则：`可预测`、`可定位`、`可修改`、`可验证`。

Agent 在没有额外说明时，应优先选择最直接、边界最清晰、影响范围最容易验证的实现方式。

---

## 1. 文件组织原则

### 1.1 目录职责必须清晰

- 业务代码按功能域组织，优先放入对应 feature 目录。
- 公共代码只放入明确的共享目录，不要随意散落。
- 脚本、配置、测试、生成产物分开存放。

推荐结构：

```text
src/
  features/
    <feature-name>/
  shared/
    components/
    utils/
    types/
tests/
scripts/
configs/
generated/
output/
```

### 1.2 优先按功能域组织，而不是按技术层平铺

优先：

```text
src/features/chat/
  ChatPage.tsx
  chat-service.ts
  chat-types.ts
  chat.test.ts
```

避免：

```text
src/components/
src/services/
src/utils/
src/misc/
```

### 1.3 新文件放到“所属功能”附近

- 修改某个功能时，优先在该功能目录内新增或修改文件。
- 只有在至少被两个独立功能复用时，才考虑抽到 `shared/`。
- 不要把暂时无法归类的代码放进 `misc`、`temp`、`common` 这类模糊目录。

---

## 2. 命名规范

### 2.1 命名必须稳定一致

- 目录名使用统一风格，默认推荐 `kebab-case`
- React 组件文件使用 `PascalCase`
- 普通模块文件使用 `kebab-case`
- 类型文件统一使用 `*.types.ts`
- 测试文件统一使用 `*.test.ts` 或 `*.test.tsx`

示例：

- `user-service.ts`
- `UserProfile.tsx`
- `auth.types.ts`
- `login-form.test.tsx`

避免混用：

- `user_service.ts`
- `UserService.ts`
- `user-service.ts`

---

## 3. 单文件与单模块规则

### 3.1 单文件只做一类事情

- 一个文件尽量只承载一个明确职责。
- 不要把 UI、数据请求、类型定义、测试辅助逻辑混在同一个大文件中。
- 超长文件应优先拆分，而不是继续堆功能。

### 3.2 单文件长度必须可控

- 单个源码文件原则上不能超过 2000 行。
- 如果文件规模接近或超过该上限，应按职责、层次或内部模块做合理拆分，而不是继续向同一文件堆叠实现。
- 拆分后的结构仍需保持入口清晰、边界稳定、验证路径明确。

### 3.3 模块边界清晰

- 每个 feature 应有明确入口。
- 对外暴露的内容集中在 `index.ts` 或明确导出文件中。
- 内部实现不要被跨目录直接依赖。

示例：

```text
src/features/auth/
  index.ts
  components/
  hooks/
  internal/
```

规则：

- 其他模块优先从 `index.ts` 引用
- 不直接引用 `internal/` 内部实现，除非任务明确要求

---

## 4. 共享代码规则

### 4.1 `shared/` 不是杂物箱

只有满足以下条件的代码才应进入共享目录：

- 被多个功能使用
- 不依赖某个具体 feature 的内部状态
- 名称和职责可以被清楚定义

### 4.2 谨慎创建 `utils`

- 可以有 `utils`，但必须按语义继续细分
- 优先使用 `date/`、`string/`、`http/`、`path/` 这类清晰分类
- 避免持续向单个 `utils.ts` 追加无关函数

---

## 5. 依赖与导入规范

### 5.1 导入路径要可读

- 在项目内统一使用一种主要导入风格
- 避免过深相对路径链条
- 避免跨层绕行引用

避免：

```ts
import x from "../../../shared/utils";
import y from "@/features/a/../../features/b";
```

### 5.2 禁止制造循环依赖

- 新增抽象前先确认不会形成双向依赖
- 如果两个模块互相依赖，优先抽出更小的共享接口或类型层

---

## 6. 测试与验证规则

### 6.1 测试位置保持一致

本项目应始终只选择一种测试布局并保持一致：

- 邻近放置：与源文件同目录
- 镜像放置：统一放在 `tests/`

不要混用两种风格。

### 6.2 每次改动都必须可验证

Agent 完成修改后必须优先运行与改动范围匹配的验证：

- 单元测试
- 类型检查
- lint
- 必要时的构建验证

如果项目缺少脚本约定，优先补充明确的验证入口，而不是依赖隐式人工检查。

---

## 7. 生成产物与敏感目录

以下目录默认视为非业务源码区：

- `generated/`
- `dist/`
- `build/`
- `output/`
- 临时导出目录

规则：

- 无明确要求时，不编辑生成产物
- 优先修改源文件而不是修改编译结果
- 临时文件不要混入业务代码目录

---

## 8. Agent 的默认决策规则

当需求没有明确指定实现位置时，按以下优先级决策：

1. 先找现有 feature 目录并在原位置修改
2. 若为该 feature 新能力，则放在该 feature 内
3. 若被多个 feature 复用，再抽到 `shared/`
4. 若属于脚本或工具链，放入 `scripts/` 或 `configs/`
5. 若只是临时分析结果，放入 `output/` 或临时目录，不混入源码

---

## 9. Agent 应避免的反模式

避免以下行为：

- 新建 `misc`、`temp`、`common` 这类边界不清目录
- 在单个文件中持续堆积多种职责
- 把业务逻辑放入名字过于泛化的 `utils.ts`
- 跨 feature 直接引用内部实现
- 修改生成产物替代修改源码
- 在没有验证的情况下结束任务

---

## 10. 最终判断标准

如果一个新加入的 Agent 能在短时间内回答以下问题，则说明结构合格：

1. 入口文件在哪
2. 某个功能应该改哪个目录
3. 公共代码该放哪
4. 测试在哪里
5. 改完运行什么验证命令
6. 哪些目录默认不应直接修改

如果做不到，就继续收紧目录职责、命名规则和模块边界。
